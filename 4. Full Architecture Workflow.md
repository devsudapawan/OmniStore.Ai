# 4. OmniStore.Ai Full Architecture Workflow

This document details exactly how data flows throughout the application and the philosophy behind its offline-first sync engine.

## 🧱 1. High-Level Architecture
The system utilizes a Clean Architecture/Feature-Driven Design approach based on an Event-Driven model:

```text
Flutter UI
    ↓
Riverpod Providers (State Management & Injection)
    ↓
Repository Layer (Business Logic & Data Bridging)
    ↓
Local Database (Isar - Source of Truth)
    ↓
Sync Engine (Background Processor)
    ↓
Supabase Cloud (Remote Storage & Backup)
```

## 🔁 2. Offline-First Philosophy
- All User Actions write to the **Local Database (Isar)** first.
- The UI immediately updates with these changes (zero latency, zero internet requirement).
- Every modified entity is marked with `sync_status = pending`.
- **Sync Engine Lifecycle:** When the app detects an internet connection, starts up, or hits a background interval, the sync engine wakes up.
  1. Compiles pending events.
  2. Resolves any conflicts (see below).
  3. Uploads local events to Supabase.
  4. Downloads external/remote updates.
  5. Updates local `sync_status` to synced.

## 💥 3. Conflict Resolution Strategy
Handling scenarios where a local device and the server both modified the same data:
- **Profile Data (Customers, Products, Settings):** Operates on a *Last-Write-Wins* policy. The most recent timestamp determines the true value.
- **Inventory & Balances:** Operates on an *Event-Merging* policy. Because we don't store "Total Stock", but rather events ("Sale: -1 Milk"), events from different sources are simply appended. The total is just a sum of all events.
- **Deletions:** Handled softly. A record is marked `is_deleted = true` to preserve foreign key constraints over the sync engine.

## 🛍️ 4. POS Sale Workflow (Core Use Case)
What exactly happens when a shopkeeper presses "Complete Sale":

1. **User Flow:**
   - Opens POS Screen.
   - Selects a Customer (or logs as a Walk-in).
   - Taps products (`CartItem`s) to add to cart, adjusts quantities, defines payment method (Cash, Udhar).
   - User hits **Complete Sale**.

2. **Backend Action Flow (Triggered via `pos_repository`):**
   - The system creates a `Transaction` row representing the sale's metadata. 
   - The system creates `TransactionItem` rows mapping the products, exact prices, and quantities to that transaction.
   - The system synthesizes `-` (negative) `InventoryEvent`s for every product sold, ensuring stock deductions.
   - If payment was made, the system writes a `PaymentEvent` which offsets the balance. If Udhar (Credit) was chosen, the transaction exists without a payment event, automatically reflecting as an outstanding debt for that customer.
   - All above records are saved simultaneously into Isar. UI updates instantly.

## 🤖 5. AI Assistant Workflow (Layered, Not Foundational)
The AI is positioned strictly as a helper, **never** an independent actor executing writes without permission.
- **Role:** Input parser, Data Analyzer.
- **Example Flow:**
  - User speaks: *"Ram bought 2 kg rice on credit."*
  - Payload is sent to the LLM backend or on-device processor.
  - LLM parses structured data: `{ Intent: "Sale", Details: { Customer: "Ram", Items: [{ Product: "Rice", Quantity: 2 }], Method: "Credit" } }`.
  - The UI presents a **Preview Card** displaying the parsed action.
  - The user manually hits **"Confirm"**, pushing the data through the standard POS Sale Workflow.
